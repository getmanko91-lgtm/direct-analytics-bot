# Спецификация приложения Direct Analytics Bot

**Версия:** 1.0  
**Дата:** 15.06.2026  
**Назначение:** внутренний сервис рекламного агентства для ежедневного мониторинга эффективности рекламных кампаний клиентов в Яндекс.Директ с доставкой сводок в Telegram.

---

## 1. Общее описание

Direct Analytics Bot — веб-приложение с фоновым планировщиком. Приложение **только читает статистику** из API Яндекс.Директ и **не изменяет** кампании, объявления, ставки, бюджеты и другие объекты рекламного кабинета.

Основные функции:
- ежедневная автоматическая выгрузка статистики за вчерашний день (с сравнением с предыдущим днём);
- учёт конверсий по выбранным целям Яндекс.Метрики;
- формирование отчёта (показы, клики, расход, CTR, CPC, конверсии, CPA с НДС 22%);
- отправка отчёта в Telegram;
- веб-интерфейс для настройки клиентов, выбора целей и ручного запуска отчётов.

---

## 2. Используемые методы API Яндекс.Директ

| Сервис | Метод / endpoint | Режим | Назначение |
|--------|------------------|-------|------------|
| **Reports** (API v5) | `POST https://api.direct.yandex.com/json/v5/reports` | Только чтение | Получение статистики `CAMPAIGN_PERFORMANCE_REPORT` за 2 календарных дня (вчера и позавчера) в разрезе кампаний |
| **Campaigns** (API v5) | `POST https://api.direct.yandex.com/json/v5/campaigns`, метод `get` | Только чтение | Получение списка ID кампаний клиента (поле `Id`) при ручной синхронизации целей |
| **Live API v4** | `POST https://api.direct.yandex.com/live/v4/json/`, метод `GetStatGoals` | Только чтение | Получение списка доступных целей Метрики по кампаниям при ручной настройке клиента |

**Не используются:** методы создания/редактирования/удаления кампаний, групп объявлений, объявлений, ключевых фраз, ставок, стратегий, баланса и прочие методы изменения данных.

Дополнительно (не API Директа): при наличии отдельного OAuth-токена с правом `metrika:read` может вызываться Management API Яндекс.Метрики для получения списка целей по ID счётчика. При недоступности Метрики приложение продолжает работу через `GetStatGoals`.

## 3. Схема последовательности вызовов

### 3.1. Ежедневный автоматический отчёт (основной сценарий)

```
Планировщик (1 раз в сутки, по умолчанию 09:00 MSK)
    │
    ├─► Для каждого активного клиента (последовательно, 1 поток):
    │       │
    │       ├─► Reports API v5: CAMPAIGN_PERFORMANCE_REPORT
    │       │     • DateRangeType: CUSTOM_DATE
    │       │     • Period: 2 дня (позавчera → вчера)
    │       │     • Goals: до 10 ID за запрос (при большем числе — несколько последовательных запросов)
    │       │     • Header Client-Login: логин кабинета клиента
    │       │     • При HTTP 201/202: ожидание retryIn секунд и повтор того же запроса
    │       │
    │       └─► Формирование текста отчёта → отправка в Telegram
    │
    └─► Завершение цикла
```

### 3.2. Ручная синхронизация целей (редкий сценарий, по действию пользователя в веб-интерфейсе)

```
Пользователь → «Синхронизировать цели»
    │
    ├─► [опционально] Metrika Management API: GET /management/v1/counter/{id}/goals
    │
    ├─► Campaigns API v5: campaigns.get (FieldNames: Id)
    │
    ├─► Live API v4: GetStatGoals (батчами до 100 CampaignIDS за запрос)
    │
    └─► Сохранение списка целей в локальную БД; пользователь отмечает нужные цели
```

### 3.3. Ручной предпросмотр / отправка отчёта

Аналогично п. 3.1, но инициируется пользователем из веб-интерфейса для одного клиента.

## 4. Частота вызовов и параллелизм

| Режим | Частота | Оценка числа запросов к API Директа |
|-------|---------|-------------------------------------|
| Автоматические отчёты | 1 раз в сутки | 14–28 запросов Reports (14 клиентов × 1–2 запроса при >10 целях) |
| Синхронизация целей | По необходимости, ~1–2 раза при первичной настройке клиента | 1× campaigns.get + 1–N× GetStatGoals на клиента |
| Ручной предпросмотр | Эпизодически | 1–2× Reports на клиента |

**Параллелизм:** 1 одновременное соединение. Клиенты обрабатываются **строго последовательно** в цикле. Многопоточные и асинхронные массовые запросы к API Директа **не применяются**.

**Средняя суточная нагрузка:** порядка 15–30 запросов Reports + единичные служебные запросы при настройке.

## 5. Объём обрабатываемых данных

| Параметр | Сейчас | План |
|----------|--------|------|
| Активных клиентов (кабинетов) | 14 | до 20 в течение 12 месяцев |
| Кампаний на клиента | зависит от кабинета, в отчёт попадают все кампании с данными за период | без существенного роста |
| Объявлений / ключевых фраз | **не запрашиваются и не обрабатываются** | не планируется |
| Целей Метрики на клиента | 1–10 (выбираются пользователем) | без изменений |
| Период в отчёте | 2 календарных дня | без изменений |

Приложение агрегирует статистику **на уровне кампаний**, детализация до объявлений и фраз не используется.

## 6. Учёт технических и балльных ограничений API

1. **Последовательная обработка клиентов** — исключение пиковых одновременных запросов.
2. **Reports offline-режим:** при ответах HTTP 201/202 приложение читает заголовок `retryIn` и повторяет запрос через указанный интервал (блокирующее ожидание), без параллельных дублирующих запросов.
3. **Лимит целей в Reports:** не более 10 `Goals` в одном запросе; при большем числе выбранных целей выполняются **несколько последовательных** запросов с последующим объединением данных.
4. **GetStatGoals:** идентификаторы кампаний передаются батчами до 100 штук.
5. **Уникальные имена отчётов:** `ReportName` содержит даты периода и unix-timestamp для снижения коллизий.
6. **Минимальный набор полей:** запрашиваются только необходимые метрики (Impressions, Clicks, Cost, Ctr, AvgCpc, Conversions, CostPerConversion по выбранным целям).
7. **Короткий период:** 2 дня — минимизация объёма выгружаемых данных.

## 7. Обработка ошибок

| Тип ошибки | Поведение приложения |
|------------|---------------------|
| HTTP 201/202 (отчёт в очереди) | Ожидание `retryIn`, повтор того же запроса |
| HTTP 400, 500, 502 | Логирование `RequestId`, ошибка пользователю, переход к следующему клиенту (в автоматическом режиме) |
| Ошибки JSON API (`error` в теле) | Отображение в веб-интерфейсе / уведомление в Telegram |
| Live API v4 (`error_code`) | Сообщение об ошибке, при частичном успехе (Метрика недоступна) — продолжение через GetStatGoals |
| Сетевые ошибки (ConnectionError) | Логирование, сообщение «Не удалось подключиться к API» |
| Ошибка одного клиента при ежедневной рассылке | Не блокирует обработку остальных клиентов; уведомление об ошибке в Telegram |

Все ошибки фиксируются в логах сервера с указанием клиента и, при наличии, `RequestId` из заголовков ответа API.

## 8. Авторизация

- OAuth-токен представителя агентства (`Authorization: Bearer`).
- Для каждого клиентского кабинета — HTTP-заголовок `Client-Login` с логином рекламодателя.
- Токен хранится на сервере в переменных окружения (`.env`), не передаётся в браузер пользователя.

---

## 9. Интерфейс приложения (экраны)

1. **Вход** — авторизация сотрудника агентства (логин/пароль).
2. **Дашборд** — список клиентов, статус, кнопка ручной отправки всех отчётов.
3. **Клиенты** — добавление/редактирование кабинета (название, логин Директа, chat_id Telegram, ID счётчика Метрики).
4. **Цели** — синхронизация списка целей, выбор целей для расчёта конверсий.
5. **Превью отчёта** — просмотр и ручная отправка в Telegram.
6. **Настройки** — время ежедневной отправки, часовой пояс.

*К заявке рекомендуется приложить скриншоты экранов «Дашборд», «Клиенты», «Цели», «Превью отчёта».*
