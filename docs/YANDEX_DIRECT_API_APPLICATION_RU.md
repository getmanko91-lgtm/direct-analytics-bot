# Описание взаимодействия приложения Direct Analytics Bot с API Яндекс.Директ

**Приложение:** Direct Analytics Bot  
**Client ID:** 4fb14971f8a74394bd68c249b3e4dea2  
**Тип доступа:** только чтение статистики (режим «Чтение» / read-only)  
**Пользователь API:** представитель рекламного агентства (OAuth-токен + заголовок Client-Login для каждого клиентского кабинета)

---

## 1. Перечень методов API Яндекс.Директ, используемых программой

Программа использует **только три метода** API Директа. Методы изменения данных (add, update, delete, suspend, resume и т.д.) **не вызываются**.

### 1.1. Сервис Reports (API v5) — формирование статистического отчёта

| Параметр | Значение |
|----------|----------|
| **Сервис** | Reports |
| **Версия API** | v5 |
| **URL** | `POST https://api.direct.yandex.com/json/v5/reports` |
| **Тип операции** | Формирование отчёта (read-only) |
| **ReportType** | `CAMPAIGN_PERFORMANCE_REPORT` |
| **DateRangeType** | `CUSTOM_DATE` |
| **Format** | `TSV` |

**Запрашиваемые поля (FieldNames):**
`Date`, `CampaignName`, `Impressions`, `Clicks`, `Cost`, `Ctr`, `AvgCpc`, `Conversions_{GoalId}`, `CostPerConversion_{GoalId}` (для каждой выбранной цели Метрики, не более 10 целей за один запрос).

**Дополнительные параметры:** `Goals` (массив ID целей Метрики, до 10 элементов), `AttributionModels` (например, `LSC`), `SelectionCriteria.DateFrom`, `SelectionCriteria.DateTo`, `IncludeVAT=NO`, `IncludeDiscount=NO`.

**HTTP-заголовки запроса:**
- `Authorization: Bearer {OAuth-token}`
- `Client-Login: {логин клиентского кабинета}`
- `Accept-Language: ru`
- `skipReportHeader: true`
- `skipReportSummary: true`
- `returnMoneyInMicros: false`

**Назначение:** получение статистики по кампаниям клиента за 2 календарных дня (вчера и позавчера) для формирования ежедневного отчёта в Telegram.

---

### 1.2. Сервис Campaigns (API v5) — метод `get`

| Параметр | Значение |
|----------|----------|
| **Сервис** | Campaigns |
| **Метод** | `get` |
| **URL** | `POST https://api.direct.yandex.com/json/v5/campaigns` |
| **Тип операции** | Чтение (read-only) |

**Тело запроса (JSON):**
```json
{
  "method": "get",
  "params": {
    "SelectionCriteria": {
      "States": ["ON", "SUSPENDED", "OFF"]
    },
    "FieldNames": ["Id"]
  }
}
```

**HTTP-заголовки:** `Authorization: Bearer`, `Client-Login`, `Content-Type: application/json`.

**Назначение:** получение списка идентификаторов кампаний клиента исключительно для последующего вызова `GetStatGoals` при первичной настройке списка целей конверсий. Поля кампаний (название, ставки, бюджет и т.д.) не запрашиваются.

---

### 1.3. Live API v4 — метод `GetStatGoals`

| Параметр | Значение |
|----------|----------|
| **Сервис** | Live API |
| **Версия** | v4 |
| **Метод** | `GetStatGoals` |
| **URL** | `POST https://api.direct.yandex.com/live/v4/json/` |
| **Тип операции** | Чтение (read-only) |

**Тело запроса (JSON):**
```json
{
  "method": "GetStatGoals",
  "param": {
    "CampaignIDS": [123456, 789012]
  }
}
```

Параметр `CampaignIDS` — массив ID кампаний, **не более 100** идентификаторов за один вызов.

**Назначение:** получение списка целей Яндекс.Метрики (GoalID, Name), доступных для кампаний клиента, чтобы пользователь мог выбрать цели для учёта конверсий в отчётах.

## 2. Схема и последовательность вызова методов

### Сценарий A. Ежедневный автоматический отчёт (основной, 1 раз в сутки)

**Инициатор:** встроенный планировщик (APScheduler), время по умолчанию 09:00 (Europe/Moscow).

**Параллелизм:** 1 поток. Клиенты обрабатываются **последовательно**, без одновременных соединений.

```
[Планировщик]
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ Цикл по 14 активным клиентам (client_1 … client_14)        │
│ Каждый клиент — строго после завершения предыдущего         │
└─────────────────────────────────────────────────────────────┘
     │
     ▼ (для каждого client_N)
┌─────────────────────────────────────────────────────────────┐
│ Шаг 1. Reports API v5                                       │
│ POST /json/v5/reports                                       │
│ ReportType: CAMPAIGN_PERFORMANCE_REPORT                     │
│ Period: DateFrom = (сегодня − 2 дня), DateTo = (сегодня − 1)│
│ Goals: до 10 ID за запрос                                   │
│                                                             │
│ Если выбрано > 10 целей:                                     │
│   Запрос 1 → Goals[1..10]  → ожидание ответа               │
│   Запрос 2 → Goals[11..20] → ожидание ответа  (редко)      │
│                                                             │
│ При HTTP 201/202: sleep(retryIn) → повтор ТОГО ЖЕ запроса  │
│ При HTTP 200:  парсинг TSV → агрегация → текст отчёта       │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 2. Отправка отчёта в Telegram (не API Директа)          │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
  Следующий client_(N+1)
```

---

### Сценарий B. Ручная синхронизация целей (по кнопке в веб-интерфейсе, эпизодически)

**Инициатор:** пользователь (сотрудник агентства). Частота: 1–2 раза на клиента при первичной настройке, затем редко.

```
[Пользователь: «Синхронизировать цели»]
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 1. Campaigns.get                                        │
│ POST /json/v5/campaigns                                     │
│ FieldNames: ["Id"]                                          │
│ → получение массива CampaignIds                               │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
┌─────────────────────────────────────────────────────────────┐
│ Шаг 2. GetStatGoals (Live v4)                               │
│ POST /live/v4/json/                                         │
│ CampaignIDS: батчами по 100 кампаний                        │
│                                                             │
│ Если кампаний 250:                                          │
│   Вызов 1 → CampaignIDS[1..100]                             │
│   Вызов 2 → CampaignIDS[101..200]                           │
│   Вызов 3 → CampaignIDS[201..250]                           │
│ (вызовы последовательные)                                   │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
[Сохранение GoalID + Name в локальную БД; выбор целей пользователем]
```

---

### Сценарий C. Ручной предпросмотр / отправка отчёта (по кнопке в веб-интерфейсе)

Идентичен **Сценарию A, Шаг 1** для одного выбранного клиента.

## 3. Частота вызова каждого метода

Расчёт для **14 клиентов**, среднее число выбранных целей — **5** (≤10, один запрос Reports на клиента).

| Метод API | Сценарий | Частота | Вызовов в сутки | Вызовов в месяц (≈30 дн.) |
|-----------|----------|---------|-----------------|---------------------------|
| **Reports** (`CAMPAIGN_PERFORMANCE_REPORT`) | Автоматический отчёт | 1 раз/сутки × 14 клиентов | **14** | **420** |
| **Reports** | Ручной предпросмотр/отправка | ~2 раза/неделю на агентство | **~0,3** | **~8** |
| **Campaigns.get** | Синхронизация целей | ~1 раз на клиента при настройке | **~0** (в штатном режиме) | **~14** (при onboarding) |
| **GetStatGoals** | Синхронизация целей | 1–3 вызова на клиента при настройке | **~0** (в штатном режиме) | **~20–40** (при onboarding) |

**Итого в установившемся режиме:** ~**14–15 вызовов Reports в сутки**, ~**420–450 в месяц**. Вызовы `Campaigns.get` и `GetStatGoals` — только при настройке новых клиентов.

**Пиковая нагрузка:** при автоматическом прогоне 14 клиентов подряд — не более **1 одновременного** HTTP-соединения к API Директа; интервал между клиентами определяется временем формирования отчёта (обычно 1–30 секунд на клиента, при offline-режиме — ожидание `retryIn`).

**Планируемый рост:** до **20 клиентов** в течение 12 месяцев → до **~20 вызовов Reports/сутки**, ~**600/месяц**.

## 4. Обработка ошибок API

| Код / ситуация | Метод | Действие программы |
|----------------|-------|-------------------|
| **HTTP 200** | Reports | Парсинг TSV, формирование отчёта |
| **HTTP 201, 202** | Reports | Чтение заголовка `retryIn` (сек.), `sleep(retryIn)`, **повтор того же запроса** без создания дубликата; логирование `RequestId` |
| **HTTP 400** | Любой | Логирование `RequestId` и тела ответа; ошибка пользователю; при автоматическом прогоне — переход к следующему клиенту |
| **HTTP 500, 502** | Reports | Логирование; сообщение «повторите позднее»; переход к следующему клиенту |
| **JSON `error` в теле** | Campaigns.get | Отображение `error_code`, `error_string`, `error_detail` пользователю |
| **`error_code` в Live v4** | GetStatGoals | Отображение ошибки; при частичном успехе — продолжение, если данные уже получены |
| **ConnectionError / timeout** | Любой | Логирование; сообщение «Не удалось подключиться к API»; timeout: Reports 120 с, остальные 60 с |
| **Ошибка одного клиента** | Автопрогон | **Не прерывает** обработку остальных 13 клиентов; уведомление об ошибке в Telegram |

Все ошибки записываются в серверный лог с указанием: клиент, метод, HTTP-код, `RequestId` (из заголовка ответа API).

## 5. Учёт ограничений API Директа

### 5.1. Ограничения сервиса Reports

1. **Offline-очередь (HTTP 201/202):** программа не создаёт параллельных запросов одного отчёта; ожидает `retryIn` и повторяет запрос.
2. **Лимит целей:** не более **10** элементов в массиве `Goals`; при большем числе — **несколько последовательных** запросов Reports с объединением результатов.
3. **Минимальный период:** запрашиваются данные только за **2 дня** (минимизация объёма).
4. **Минимальный набор полей:** только метрики, необходимые для отчёта; без лишних срезов и фильтров.
5. **Уникальное имя отчёта:** `ReportName = DailyAnalytics_{DateFrom}_{DateTo}_{unix_timestamp}` — исключение коллизий имён.

### 5.2. Ограничения Campaigns.get

1. Запрашивается **только поле `Id`** — минимальный расход баллов/ресурсов.
2. `SelectionCriteria.States` ограничен тремя статусами; пагинация не требуется при типичном числе кампаний агентских клиентов.

### 5.3. Ограничения GetStatGoals (Live v4)

1. Не более **100** ID кампаний в `CampaignIDS` за вызов; при большем числе — **последовательные батчи**.
2. Вызов только при ручной настройке, не в ежедневном цикле.

### 5.4. Общие меры (баллы, параллелизм, частота)

1. **1 одновременное соединение** — исключение конкурентных запросов от одного OAuth-токена.
2. **Последовательная обработка 14 клиентов** — равномерное распределение нагрузки в окне ~09:00–09:30.
3. **Отсутствие polling-циклов** — нет периодического опроса API чаще 1 раза в сутки на клиента.
4. **Отсутствие методов записи** — программа не расходует баллы на изменение объектов.
5. **Кэширование целей** — список целей хранится в локальной БД; повторный `GetStatGoals` только по явному действию пользователя.
